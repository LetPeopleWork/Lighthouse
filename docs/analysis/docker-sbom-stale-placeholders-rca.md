# Root Cause Analysis: Docker SBOM Shows Stale Placeholder Packages

**Date**: 2026-07-19
**Severity**: P1 — Feature regression (third-party package visibility broken for Docker users)
**Scope**: Docker packaging pipeline only; standalone/server unaffected

---

## Problem Statement

Docker users of Lighthouse see only 2 stale placeholder third-party packages instead of the full, CI-generated SBOM list. Standalone and server (non-Docker) versions display all packages correctly.

---

## 5-Whys Causal Chain

### Branch A: Docker build ships stale SBOM files

```
WHY 1A: Docker image serves only 2 placeholder packages, not the full SBOM
  [Evidence: User reports; Lighthouse.Frontend/src/services/Api/SystemInfoService.ts:58-81
   fetches /sbom/backend.cdx.json and /sbom/frontend.cdx.json; committed files at
   Lighthouse.Frontend/public/sbom/ contain only 1-2 components each]

  WHY 2A: The Vite Docker build (`pnpm run build-docker`) copies committed SBOM
          placeholder files from `public/sbom/` into the final `dist/` directory
    [Evidence: Dockerfile:26-30 copies Lighthouse.Frontend source and runs
     `build-docker`; Vite copies `public/` → `dist/` during build; committed
     backend.cdx.json and frontend.cdx.json are stale placeholders]

    WHY 3A: The Docker CI workflow has no step to download real, CI-generated
            SBOM artifacts and replace the committed placeholders before building
      [Evidence: ci_docker.yml contains no SBOM artifact download step; standalone
       workflows DO have this step — e.g., ci_package-linux-standalone.yml:53-63
       downloads SBOM artifacts and copies them to public/sbom/ before build]

      WHY 4A: The Docker packaging workflow was never updated when SBOM generation
              was added to CI, because SBOM was scoped only to standalone packaging
        [Evidence: All 3 standalone packaging workflows received SBOM download
         steps (linux, macos, windows). Docker workflow (ci_docker.yml) has no
         corresponding update. Real SBOMs are generated by ci_sbom.yml using
         dotnet-CycloneDX + @cyclonedx/cdxgen — but Docker never consumes them.]

        WHY 5A: No cross-packaging parity mechanism exists — packaging workflows
                independently duplicate their build logic without a shared spec
                or automated consistency gate that would flag missing SBOM
                integration when a new packaging target is added or modified
        → ROOT CAUSE A: Absence of packaging-workflow parity validation

        WHY 5B (alternate branch): Committed SBOM placeholders mask the problem
               at build time — the build succeeds with incorrect data, producing
               no alarm. If SBOM files were absent (causing build failure) rather
               than containing stale data, Docker builds would have failed and
               forced resolution at integration time
        → ROOT CAUSE B: Silent staleness — placeholder files defeat build-time correctness checks
```

---

## Root Causes

| # | Root Cause | Category |
|---|-----------|----------|
| **A** | No cross-packaging parity validation — each packaging workflow independently copies build steps without automated consistency checks. SBOM integration was added to 3 standalone workflows but the 4th (Docker) was silently omitted. | Process / CI Design |
| **B** | Committed SBOM placeholders mask the problem — build succeeds with incorrect data instead of failing. The absence of a "minimum components" assertion or file-size check allows stale data to ship undetected. | Build-time Validation |

---

## Contributing Factors

| # | Factor | Impact |
|---|--------|--------|
| C1 | No integration test validates third-party package display in the Docker image | Regression undetected until users report it |
| C2 | SBOM generation and consumption logic is duplicated per workflow (copy-paste) rather than centralized in a reusable action or shared CI template | Inconsistency inevitable as workflows diverge |
| C3 | The `docker` job in `ci.yml` only depends on `version` — no `needs: sbom` dependency to signal the relationship | CI graph doesn't model the data dependency |

---

## Cross-Validation

- **Root Cause A → Symptom**: If no parity validation exists, adding a feature to 3 of 4 packaging targets produces exactly the observed gap. ✓
- **Root Cause B → Symptom**: If committed SBOMs were absent (not stale), the Docker build would have failed with missing-file errors, forcing the SBOM step to be added. The presence of placeholders allowed the gap to go undetected. ✓
- **A + B consistency**: Both independently explain the same gap from different angles (process vs. safety net). They reinforce rather than contradict. ✓
- **All symptoms explained**: The stale placeholder packages observed by Docker users are fully explained by the Docker build never receiving real SBOM files (A) and the build succeeding silently with stale placeholders (B). ✓

## Proposed Fix

### Permanent Fix (addresses both root causes)

#### Fix 1: Add SBOM download to Docker workflow

Add the same SBOM artifact download + copy pattern to the Docker build pipeline, matching the standalone workflow implementation.

**Files affected**:
- `ci_docker.yml` — add SBOM download step
- `ci.yml` — add `sbom` to Docker job's `needs` array

**Pattern** (from `ci_package-linux-standalone.yml:53-63`):
```yaml
- name: Download SBOM artifacts
  uses: actions/download-artifact@v4
  with:
    name: sbom
    path: ./Lighthouse.Frontend/public/sbom/
```

In `ci.yml`:
```yaml
# Change Docker job's needs from:
needs: [version]
# To:
needs: [version, sbom]
```

#### Fix 2: Replace committed placeholders with build-time assertion

Remove the committed stale SBOM files and add a build-step validation that verifies SBOM files have a reasonable number of components (> 5). This ensures any future packaging target omission fails loudly at build time rather than silently shipping stale data.

**Files affected**:
- `Lighthouse.Frontend/public/sbom/backend.cdx.json` — delete placeholder
- `Lighthouse.Frontend/public/sbom/frontend.cdx.json` — delete placeholder
- `Dockerfile` or build script — add validation step

**Validation script** (pseudocode):
```bash
# Validate SBOM has real content, not a placeholder
for f in Lighthouse.Frontend/public/sbom/*.cdx.json; do
  count=$(jq '.components | length' "$f")
  if [ "$count" -lt 5 ]; then
    echo "ERROR: $f has only $count components — SBOM generation step may be missing"
    exit 1
  fi
done
```

### Early Detection (operational improvement)

- Add a CI step or E2E smoke test that starts the Docker container and asserts the `/sbom/` endpoints return > N components.

## Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|------------|
| Deleting placeholder SBOMs breaks local dev builds | Low | Local dev builds don't need real SBOMs; can use mock data or skip validation in dev mode. Add `.gitkeep` + dev-mode bypass for the validation check. |
| SBOM artifact download adds ~10s to Docker CI pipeline | Negligible | Artifact download is fast; outweighed by correctness benefit. |
| Validation check false-positive if component count threshold is wrong | Low | Set threshold conservatively (> 3); real SBOMs have 50+ components for any non-trivial project. |
| Fix only addresses Docker — other future packaging targets still at risk | Medium | Root Cause A (parity validation) not fully addressed by this fix alone. Follow up with centralized SBOM consumption action. |

### Follow-up Recommendation

Extract the SBOM download-and-copy pattern into a reusable GitHub Actions composite action used by all 4+ packaging workflows. This directly addresses Root Cause A by making it impossible to add a new packaging target without also inheriting the SBOM step.
