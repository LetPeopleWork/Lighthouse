# ADR-083: Publish the Public Helm Repo From the Existing Repo's Docs Tree (`docs/charts/`) via the Current Artifact-Based GitHub Pages Deploy, in the Existing Release Stage — NOT `gh-pages` / chart-releaser; Chart.yaml `version` Is the Single Source of Truth, Guarded Against Silent Overwrite

**Status**: **Accepted** (2026-06-21 — accepted by Benjamin)
**Date**: 2026-06-21
**Feature**: epic-5306-k8s-productization (ADO Epic #5306, story #5199)
**Decider**: Benjamin (product owner) + System Designer (PROPOSE)
**Relationship to prior work**: Realises planning-stage **D1** ("the chart is public and lives in this repo, published to a Helm repo via GitHub Pages"). Honours the CLAUDE.md CI-consolidation rule (extend existing workflows, never add a new one).

---

## Context

DISCUSS locked distribution to "a public OSS chart on a GitHub Pages Helm repo (LPW org)". The user's stated priorities: keep the chart in the **existing** `lighthouse` repo, and **reuse the existing release stage** so chart and app releases stay aligned.

A concrete infrastructure fact constrains the mechanism: the repo's docs are **already** on GitHub Pages via the **modern artifact-based deploy** — `.github/workflows/pages.yml` builds Jekyll from `docs/` (`_site`), uploads it with `actions/upload-pages-artifact`, and publishes with `actions/deploy-pages`, behind a CNAME. **GitHub Pages allows exactly one publishing source per repo.** The conventional Helm-publish tool, `helm/chart-releaser-action`, pushes `.tgz` + `index.yaml` to a **`gh-pages` branch** — a *different, mutually exclusive* Pages source. So chart-releaser and the existing docs deploy cannot coexist in one repo. The earlier preference for chart-releaser is therefore incompatible with the (preferred) same-repo outcome.

## Decision

**Publish the Helm repo as static files inside the docs tree the existing Pages deploy already serves. Do not use a `gh-pages` branch and do not use chart-releaser-action.**

- The Helm repo lives at **`docs/charts/`** → served by the existing `pages.yml` at **`https://docs.lighthouse.letpeople.work/charts`** (the repo's GitHub Pages CNAME is `docs.lighthouse.letpeople.work`; this is the value of `helm repo add letpeoplework https://docs.lighthouse.letpeople.work/charts`).
- A step in the **existing release stage** runs `helm package chart/ -d docs/charts/` then `helm repo index docs/charts/ --merge docs/charts/index.yaml --url https://docs.lighthouse.letpeople.work/charts`, commits the new `.tgz` + updated `index.yaml`, and lets `pages.yml` (which already triggers on `docs/**`) publish it. App release and chart publish are one aligned flow.
- **No-silent-overwrite guard (the finalization gate):** before packaging, a check asserts the `Chart.yaml` `version` is **not already present** in `docs/charts/index.yaml`. If it is, the release step fails loud ("chart version X already published — bump Chart.yaml version") rather than overwriting an existing artifact. This is the "refuse silent overwrite" AC, owned by us in ~15 lines (the capability chart-releaser would have provided for free, now explicit because chart-releaser is out).
- **`Chart.yaml: version` is the single source of truth** for the chart version. Everything else derives: NOTES.txt prints `.Chart.Version`; the README install snippet is generated by helm-docs from Chart.yaml (ADR-084); `index.yaml` is built from the packaged Chart.yaml. `Chart.yaml: appVersion` mirrors the default `image.tag` (the Lighthouse image the chart ships), and a consistency check asserts `appVersion == values.image.tag` so the chart never points at a stale image. With the release stage reused, `version` bumps alongside each app release.

## Consequences

- **Positive**: one repo (the user's preference); one Pages source (no conflict with the working docs deploy); chart and app releases aligned through the existing release stage; the existing `pages.yml` `docs/**` trigger publishes the index with no new workflow (CI-consolidation honoured); version consistency and no-overwrite are explicit, testable checks.
- **Negative / cost**: we maintain the ~15-line package+index+guard step instead of the prebuilt chart-releaser-action; `.tgz` artifacts are committed under `docs/charts/` (Git grows by chart size per release — small).
- **Standalone gate**: N/A (publishing concern, no runtime code).

## Alternatives considered

1. **`helm/chart-releaser-action` + `gh-pages` branch (same repo).** Rejected — a `gh-pages` branch is a second Pages source; GitHub Pages serves only one per repo, so it would break the existing artifact-based docs deploy.
2. **Dedicated `letpeoplework/helm-charts` repo with chart-releaser + gh-pages.** Viable and clean, but rejected per the user's same-repo preference; it would also de-align chart releases from the app release stage (separate workflows). Recorded as the fallback if the docs-tree approach ever strains.
3. **Helm OCI registry (GHCR) instead of an HTTP Helm repo.** Rejected — DISCUSS locked distribution to a GitHub Pages HTTP Helm repo; OCI changes the `helm repo add` UX the stories specify.
