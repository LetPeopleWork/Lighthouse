# CI Learnings

Durable rules derived from CI / SonarCloud failures on `Build And Deploy Lighthouse`. Append a new entry every time `/clean-ci` resolves a failure. Read this file before touching code in the related area.

Each entry follows:

```
### YYYY-MM-DD ‚Äî <short title>
- **Symptom**: what CI / Sonar reported (rule key, error excerpt, job name).
- **Root cause**: the actual reason, in one sentence.
- **Fix**: what was changed (file:line is enough; the commit has the diff).
- **Rule going forward**: a single declarative do/don't sentence future-Claude can apply BEFORE writing similar code.
```

## Automated pre-commit checks (machine-readable)

`Scripts/ledger_check.py` runs as a `PreToolUse` hook on `git commit` (wired in
`.claude/settings.json`) and greps the **staged** diff against the patterns below. A match
**hard-blocks the commit** ‚Äî the hook's message is fed back to Claude, which fixes the line,
re-stages, and commits again ‚Äî so a known foot-gun never enters git history. Add a line
here (same delimiter) whenever `/clean-ci` records a learning whose foot-gun is a
deterministic, low-false-positive line pattern. Format (4 fields, ` ::: ` delimited):

`<rule-key> ::: <comma-separated file extensions> ::: <Python regex> ::: <fix hint>`

<!-- LEDGER-CHECKS:START -->
typescript:S7735 ::: ts,tsx ::: (!==|!=)\s*(undefined|null)\s*\? ::: Negated condition in a ternary ‚Äî flip to `X === undefined ? falsy : truthy` (no `!==`/`!=` in a ternary condition). See the 2026-05-16 S7735 entry.
dotnet-nunit:NUnit4002 ::: cs ::: Is\.EqualTo\(0\) ::: Use `Is.Zero`, never `Is.EqualTo(0)` (NUnit4002).
external_roslyn:CA1861 ::: cs ::: (Is|Does|Has)\.\w+\(new\[\] ::: CA1861 ‚Äî extract the inline `new[] {...}` assertion array to a `private static readonly` field; Roslyn flags constant array args passed to repeatedly-called methods.
<!-- LEDGER-CHECKS:END -->

## Formatting & linting

### 2026-06-18 ‚Äî MUI bump fails Vitest with `ERR_UNSUPPORTED_DIR_IMPORT` even though `pnpm build` is green
- **Symptom**: `Verify Frontend / frontend` failed on a Dependabot `@mui/material` 9.0.1‚Üí9.1.1 bump. 5 test files (the date-picker / forecaster ones that `vi.mock("@mui/material")` **and** `vi.mock("@mui/x-date-pickers")`) errored: `[vitest] There was an error when mocking a module` caused by `Error: Directory import '.../react-transition-group/TransitionGroupContext' is not supported ‚Ä¶ ERR_UNSUPPORTED_DIR_IMPORT` from `@mui/material/internal/Transition.mjs`. `pnpm build` (tsc + vite/rollup) passed cleanly.
- **Root cause**: MUI 9.1.x's `Transition.mjs` deep-imports `react-transition-group/TransitionGroupContext`, which is a *directory* whose `package.json` only has `main`/`module` redirects. Vite/Rollup honour those (build passes), but Node's native ESM resolver ‚Äî which Vitest uses for any **non-inlined** dep, including everything pulled in by `vi.importActual` ‚Äî rejects directory imports. A first fix attempt added the plain string `"@mui/material"` to `server.deps.inline`; it did **not** work because (a) Vitest string-inline didn't force the deep `.mjs` subpath through Vite, and (b) the same tests also `importActual` `@mui/x-date-pickers`, which independently pulls the transition.
- **Fix**: `Lighthouse.Frontend/vitest.config.ts` ‚Äî `server.deps.inline: [/@mui\//, /react-transition-group/]` (regex, not strings). Regex matches every `@mui/*` subpath and `react-transition-group` so Vite transforms them and resolves the directory import. Verified full suite 264 files / 3474 tests green.
- **Rule going forward**: When a Vitest run dies with `ERR_UNSUPPORTED_DIR_IMPORT` from a node_modules dep after a dependency bump, add that package (and the dep it directory-imports) to `server.deps.inline` as a **regex** (`/pkg/`), not a bare string ‚Äî and remember `pnpm build` can't catch it because Rollup tolerates the directory import that Node ESM forbids.

### 2026-05-12 ‚Äî SonarCloud's PowerShell analyzer is strict; new `.ps1` files trigger many rules
- **Symptom**: Sonar gate failed with 12 new violations on `LetPeopleWork_Lighthouse` after a fresh `.ps1` file was committed. Rules fired: `powershelldre:S8620` (trailing whitespace), `powershelldre:S8677` (`Write-Host` should be `Write-Output` for pipelining), `powershelldre:S8641` (`$null` must be on the LEFT of comparisons), `powershelldre:S8642` (cmdlet-style casing on identifiers).
- **Root cause**: PowerShell files are linted by Sonar with stricter conventions than ad-hoc shell scripting habits. `Write-Host` discards its output (un-pipelable) and is the de-facto debug-print habit; PowerShell-strict code uses `Write-Output` for informational text and reserves `Write-Host` for things that genuinely need TTY-only behaviour (e.g. `-ForegroundColor`). Comparing `$variable -eq $null` is unreliable in PowerShell because of array-flattening semantics; `$null -eq $variable` is the safe order.
- **Fix**: Trailing whitespace removed. `Write-Host "info"` ‚Üí `Write-Output "info"` where colour wasn't needed. `Write-Host "..." -ForegroundColor Green` was kept as-is (intentional terminal-only). `$null -eq (Get-Process ...)` ordering flipped. The cmdlet-casing rule firing on the literal `pkcs11-tool` (an external binary, not a PowerShell cmdlet) was worked around by storing the name in a variable and invoking via `& $varname`.
- **Rule going forward**: In any new `.ps1` file: (a) prefer `Write-Output` for non-coloured informational text ‚Äî only use `Write-Host` when `-ForegroundColor` is essential; (b) write `$null -eq $foo`, never `$foo -eq $null`; (c) trim trailing whitespace; (d) if you need to call an external binary whose name looks like a cmdlet (e.g. contains a hyphen), store the name in a variable and invoke with `& $name args...` to bypass the cmdlet-casing analyzer.

### 2026-06-09 ‚Äî CA1859 / CA1861 surface only in the SonarCloud backend gate, not the local build

- **Symptom**: Backend Sonar gate failed with 5 new violations ‚Äî `external_roslyn:CA1859` (private methods returning `IReadOnlyList<T>`/`IReadOnlyCollection<T>` that actually build and return a `List<T>`) and `external_roslyn:CA1861` (inline `new[] { ... }` arrays passed to NUnit assertions like `Is.EquivalentTo(new[] {...})`). A clean local `dotnet build` (TreatWarningsAsErrors) passed ‚Äî these CA rules sit at suggestion/info severity locally, so they never break the build; they only fail Sonar's `new_violations=0` gate.
- **Root cause**: CA1859 wants concrete return types on non-public methods for perf (the JIT can devirtualise); CA1861 wants constant arrays hoisted to `static readonly` fields rather than reconstructed per call. Both are in the CLAUDE.md recurring list but are easy to re-introduce because the local build is silent on them.
- **Fix**: CA1859 ‚Äî change the private/internal method's declared return type from the interface to the concrete `List<T>` it returns. CA1861 ‚Äî extract the inline `new[] {...}` to a `private static readonly T[] Foo = [...]` field and reference it.
- **Rule going forward**: For any NEW non-public method that returns `.ToList()`/`.ToArray()`, declare the return type as the concrete `List<T>`/`T[]`, not `IReadOnlyList<T>`/`IReadOnlyCollection<T>`. Never pass an inline `new[] {...}` to an NUnit assertion ‚Äî hoist it to a `static readonly` field. The pre-commit hook now greps the CA1861 assertion form; CA1859 (return type) is not line-greppable, so apply it by habit.
- **Recurrence: 1** ‚Äî 2026-06-09, multiple-cycle-times step 04-02 (`NamedCycleTimeCumulativeScopeIntegrationTest.cs:187`): a NEW test helper `StateNames` declared `IReadOnlyList<string>` but `return ‚Ä¶ToList();`. Sonar fired CA1859 (1 new violation ‚Üí backend gate ERROR) while a sibling NEW production helper `ScopedCumulativeStateOrder` (also `IReadOnlyList<string>` via `.ToList()`) did NOT fire ‚Äî its result is immediately consumed as an `IReadOnlyList<T>` parameter, so the concrete type buys nothing and CA1859 stays silent. **Sharpening**: CA1859 fires when the result is consumed by something that takes `object`/no-better-type (NUnit `Does.Contain`/`Does.Not.Contain` constraints, here) ‚Äî the interface-vs-concrete decision depends on the CALL SITE, not just the method. Test helpers feeding NUnit constraints should return the concrete `List<T>`/`T[]`. Test code is analysed; the gate is `new_violations = 0`.
- **Recurrence: 2** ‚Äî 2026-06-09, multiple-cycle-times mutation gate (`CycleTimeDefinitionValidatorTest.cs:9`): a NEW test factory `SettingsWith(...)` declared the base `SettingsOwnerDtoBase` return but always `new TeamSettingDto { ... }`. CA1859 fired even though the consumer (`ValidateSettings`) takes the base ‚Äî Roslyn flags the **declared-vs-constructed** mismatch on the helper itself. **Rule reinforced**: a private/test factory's return type must be the CONCRETE type it `new`s, never an abstract base or interface, regardless of how the immediate caller consumes it. Pre-flight every new `private static` factory: if the body is a single `new Concrete{...}`/`...ToList()`, the return type IS that concrete type.
- **Recurrence: 3** ‚Äî 2026-06-14, bundled demo-data commit (`DemoDataFactoryTest.cs:140`): a NEW private helper `ParseColumnDates` declared `IReadOnlyList<DateTime>` but `return ‚Ä¶ToList();`; consumed only by NUnit `Is.Not.Empty`/`Has.None.Matches` constraints (no better type), so CA1859 fired ‚Üí backend `new_violations=1` ‚Üí gate ERROR while local `dotnet build` was clean. Fix: return `List<DateTime>`. Same single-line class as before; surfaced only because the demo-data commit had never been through CI.

## Build & compile

### 2026-06-16 ‚Äî pnpm settings (`overrides`, `onlyBuiltDependencies`) must live in `pnpm-workspace.yaml`, not `package.json`, or CI frozen-install fails (Recurrence: 1)
- **Recurrence 2026-06-18**: a Dependabot fix added `pnpm.overrides` (undici, @babel/core) to `Lighthouse.Frontend/package.json` AND a new `pnpm.overrides` (form-data) to `Lighthouse.EndToEndTests/package.json` ‚Äî both `Verify Frontend` and `Verify End to End Tests` then died with the same `ERR_PNPM_LOCKFILE_CONFIG_MISMATCH`. Worse, the frontend `package.json#pnpm.overrides` *clobbered* the existing `{ws, form-data}` entries the lockfile inherited from `pnpm-workspace.yaml`. Fix: moved undici + @babel/core into the frontend `pnpm-workspace.yaml`; created a NEW `Lighthouse.EndToEndTests/pnpm-workspace.yaml` for the form-data override; deleted both `package.json#pnpm` blocks; regenerated both lockfiles. **This entry already existed and was not consulted before editing ‚Äî read `docs/ci-learnings.md` BEFORE touching any `package.json`/lockfile.**
- **Symptom**: `Verify Frontend` + `Package App` (and `Verify End to End Tests` on recurrence) failed at `pnpm install --frozen-lockfile` with `ERR_PNPM_LOCKFILE_CONFIG_MISMATCH ‚Äî The current "overrides" configuration doesn't match the value found in the lockfile`. Reproduced ONLY in CI / a `node:24` container, never locally.
- **Root cause**: CI's runner (node 24) resolves pnpm via **corepack ‚Üí pnpm 11.x**, which **ignores the `pnpm` field in `package.json`** (`pnpm.overrides`, `pnpm.onlyBuiltDependencies`) ‚Äî it prints `[WARN] The "pnpm" field in package.json is no longer read by pnpm`. So pnpm 11 sees *zero* overrides while the lockfile records `{ws, form-data}` ‚Üí frozen mismatch. Local dev (node 26, no corepack) ran real pnpm 10.33.2, which *does* read `package.json`, so it passed ‚Äî masking the bug. Same pnpm "version" string, different effective binary.
- **Fix**: move `overrides` and `onlyBuiltDependencies` out of `package.json#pnpm` into `Lighthouse.Frontend/pnpm-workspace.yaml` (the new canonical home, read by both pnpm 10.x and 11.x). The lockfile is unchanged (its `overrides:` section was already correct); only the *source* moved. Verified frozen-install + `pnpm build` pass in a `node:24` container under both pnpm 10.33.2 and corepack's 11.7.0.
- **Rule going forward**: never put pnpm settings (`overrides`, `onlyBuiltDependencies`, `packageExtensions`, `patchedDependencies`, etc.) in `package.json#pnpm` ‚Äî they belong in `pnpm-workspace.yaml`. To reproduce a frontend CI failure that won't repro locally, run the exact step in a `node:24` container (`docker run --rm -e CI=true -v "$PWD/Lighthouse.Frontend":/app -w /app node:24 ‚Ä¶`) ‚Äî local node/pnpm can differ from CI's corepack-resolved pnpm.


### 2026-05-16 ‚Äî ASP.NET Core [FromQuery] params are required by default under nullable-enabled C#

- **Symptom**: `OAuthController.Callback` returned 400 `"The code field is required."` when the user cancelled at the Microsoft consent screen. Microsoft redirects to `/api/oauth/callback?error=access_denied&state=...` ‚Äî WITHOUT a `code` param. The controller's try/catch around `oauthService.CompleteAsync` never gets to run because the model binder rejects the request first.
- **Root cause**: Under `<Nullable>enable</Nullable>` (which the project enforces project-wide), non-nullable reference types on action parameters are treated as REQUIRED by ASP.NET's model binder. `[FromQuery] string code` ‚áí "must be present and non-null"; missing ‚Üí 400 with `ValidationProblemDetails` before the action body runs. There's no exception to catch in the controller because the controller never runs.
- **Fix**: Declare optional query params nullable explicitly: `[FromQuery] string? code`. Handle the missing-param case in the action body (redirect, BadRequest, etc., as the semantics require).
- **Rule going forward**: For any `[FromQuery]` / `[FromRoute]` / `[FromHeader]` parameter that the controller wants to inspect-and-handle-when-missing (not just consume-or-fail), declare the parameter nullable (`string?`, `int?`, etc.) so the model binder doesn't 400 before your action body runs. The default required-by-default behaviour is correct for "must be present" params (e.g. `id` on a `GET /widgets/{id}` route), but wrong for OAuth callbacks where the spec allows either a `code` OR an `error` parameter.

### 2026-05-17 ‚Äî Azure DevOps requires interactive sign-in to materialise Entra identities before API calls work

- **Symptom**: After a successful OAuth connection in Lighthouse, the first ADO API call (e.g. `GetAllProjects` while loading boards for a new team) throws `Microsoft.VisualStudio.Services.Common.VssServiceException: Identity {guid} has not been materialized, please use interactive login over the browser first.` The error originates from `VssConnection.GetClient<T>()` ‚Üí first call hits `GetConnectionDataAsync` which validates the OAuth token's principal against ADO's internal user table. The token IS valid; the principal isn't recognised.
- **Root cause**: Microsoft Entra ID is the identity provider; Azure DevOps is a separate service that maintains its own user-identity table. ADO does NOT auto-create user records from token claims ‚Äî it requires an explicit one-time interactive browser sign-in to materialise the Entra principal in its user table. This is by design (prevents API-only enumeration by unmaterialised principals) but invisible to OAuth-only integrations until the first API call fires.
- **Fix**: Operator-side, not code: navigate to `https://dev.azure.com/{org}` in a browser, sign in with the same Entra account that OAuth'd, accept any "request access" prompt for org membership, wait for the home page to load. Subsequent OAuth-authenticated API calls succeed because the principal is now materialised. Doc updated with the troubleshooting row pointing operators at the fix; no Lighthouse code change can avoid this ‚Äî ADO refuses to materialise via the API.
- **Rule going forward**: For any integration that authenticates to Azure DevOps via Entra ID OAuth (rather than a PAT issued from within the ADO UI), the connector-admin onboarding flow MUST include a step "sign into `https://dev.azure.com/{org}` in a browser at least once with the OAuth account" BEFORE the first API call. Do not skip this in customer-facing setup docs even if it sounds obvious ‚Äî it's a discoverable-only-on-first-error gotcha that bites every customer once. PATs do not have this requirement because they are minted from within the ADO UI by an already-materialised user.

### 2026-05-17 ‚Äî Microsoft Identity Platform v2.0 requires resource-prefixed scopes (AADSTS650053)

- **Symptom**: ADO OAuth connect surfaced `AADSTS650053: The application 'Lighthouse - Test' asked for scope 'vso.work_write' that doesn't exist on the resource '00000003-0000-0000-c000-000000000000'.` The user-visible error was `invalid_client` in the Lighthouse popup-complete page; the AADSTS code came out of the backend `oauth.callback.idp_error` log only after the `error_description` capture was added.
- **Root cause**: Under the Microsoft identity platform v2.0 endpoints (`login.microsoftonline.com/{tenant}/oauth2/v2.0/{authorize,token}`), scope strings MUST be **resource-prefixed** with either the resource's Application ID GUID or its registered URI. When no prefix is given, Microsoft defaults the audience to **Microsoft Graph** (resource GUID `00000003-0000-0000-c000-000000000000`). Graph has no `vso.*` scopes, so the request fails with AADSTS650053. The legacy `app.vssps.visualstudio.com/oauth2/...` flow accepted unprefixed `vso.*` strings; the v2.0 flow does not. Azure DevOps's resource GUID is `499b84ac-1321-427f-aa17-267ca6975798`; the correct scope for Lighthouse to send is `499b84ac-1321-427f-aa17-267ca6975798/vso.work_write` (plus `offline_access` unprefixed for the refresh token, which is a global OIDC scope rather than per-resource).
- **Fix**: `AdoOAuthProvider.cs` ‚Äî extract `AzureDevOpsResourceId` constant, build `AzureDevOpsDefaultScopes` as `[AzureDevOpsResourceId + "/vso.work_write", "offline_access"]`. Tests updated to assert the resource-prefixed scope in the authorize-URL query string and the token-endpoint form body.
- **Rule going forward**: For any OAuth provider that uses Microsoft Identity Platform v2.0 endpoints (`login.microsoftonline.com/.../oauth2/v2.0/...`), every resource-specific scope MUST be prefixed with either the resource's Application ID GUID (`{guid}/{scope-name}`) or its registered URI (`https://{resource}/{scope-name}`). Unprefixed scopes default to Microsoft Graph as the audience, which almost always fails for non-Graph resources. The legacy unprefixed `vso.*` form belongs to the deprecated `app.vssps.visualstudio.com/oauth2/...` flow and does not work with v2.0 endpoints. `offline_access` is the exception ‚Äî it's an OIDC global scope and stays unprefixed.

## Runtime / startup

### 2026-05-13 ‚Äî `Authentication__AllowedOrigins` scalar env var bound to empty list, crashing host
- **Symptom**: Deployed Lighthouse container failed to start with `System.InvalidOperationException: Authentication is enabled but Authentication:AllowedOrigins is empty.` thrown by `Program.EnsureCorsFailsClosed`. Operator's container set `Authentication__AllowedOrigins=https://localhost:48332` (scalar, no `__N` suffix) ‚Äî the natural form, and the form the repo's own `render.yaml:57` template prescribes.
- **Root cause**: `AuthenticationConfiguration.AllowedOrigins` is `IReadOnlyList<string>`. .NET's environment-variable configuration provider binds list-typed properties ONLY via indexed keys (`Authentication__AllowedOrigins__0`, `__1`, ...); a scalar `Authentication__AllowedOrigins=value` silently binds to an empty list. The fail-closed CORS guard then refused to start.
- **Fix**: `Lighthouse.Backend/Program.cs` ‚Äî added `LoadAuthenticationConfiguration(WebApplicationBuilder)` helper that binds via `Get<AuthenticationConfiguration>()` first, then if `AllowedOrigins` is empty falls back to the scalar configuration key and splits on `,`/`;` (trim entries, drop empties), returning a new record via `with`-expression. Both `EnsureCorsFailsClosed` and `ConfigureCors` consume the helper; the indexed form remains correct because the initial bind populates the list and the fallback path is skipped.
- **Rule going forward**: ASP.NET Core list-typed configuration (`IReadOnlyList<string>` / `List<string>` / `string[]`) populated from environment variables MUST accept both the indexed form (`Section__Key__0`, `__1`, ...) AND a scalar comma- or semicolon-separated form, with a binding-layer fallback that converts the scalar to a list. The indexed-only contract is a foot-gun for operators because it differs from the JSON/appsettings shape ‚Äî never publish a deployment template using the scalar form unless the binding accepts it.

### 2026-05-17 ‚Äî OAuth callback origin must be in `Authentication:AllowedOrigins` for reverse-proxy deployments
- **Symptom**: Operator deploys Lighthouse behind a reverse proxy (nginx / Traefik / Caddy / cloud LB) at a public hostname like `https://lighthouse.example.com`, configures `Authentication__AllowedOrigins` correctly for the UI origin they use internally (e.g. the container's `Lighthouse__BaseUrl`), and starts an OAuth connect flow. The IdP authorizes the user and redirects the browser to `https://lighthouse.example.com/api/oauth/callback?...`. The flow fails: the browser sees a CORS-blocked response, the popup-complete page never renders, and nothing actionable is logged at the `OAuthController` layer ‚Äî the failure is upstream of the controller because the request is rejected by `Program.EnsureCorsFailsClosed` before routing.
- **Root cause**: The OAuth callback at `GET /api/oauth/callback` sits behind the SAME `EnsureCorsFailsClosed` guard from the 2026-05-13 entry (see above on list-typed env-var binding) ‚Äî but the trigger is different. Direct UI traffic and IdP-driven browser redirects share the guard; if the public, browser-facing origin of the Lighthouse UI is not in `Authentication:AllowedOrigins`, the IdP redirect arrives with an `Origin` header the guard doesn't recognise and is fail-closed-rejected. The internal `Lighthouse__BaseUrl` (what the container thinks its URL is) is NOT enough ‚Äî the browser uses the public hostname, and that's what must be allow-listed.
- **Fix**: Operator-side, no Lighthouse code change. Add the public OAuth callback origin to `Authentication__AllowedOrigins` ‚Äî indexed form `Authentication__AllowedOrigins__0=https://lighthouse.example.com` (preferred), or the scalar comma/semicolon form per the 2026-05-13 fix. Stub-mode in CI (`http://localhost:5000`) avoids this entirely because the UI origin and the OAuth callback origin are the same host.
- **Rule going forward**: Any reverse-proxy, preview-deploy, or dev-tunnel deployment of Lighthouse MUST include the *public* (browser-facing) origin in `Authentication:AllowedOrigins`, not just the internal container URL. The IdP-driven callback redirect path uses the same CORS fail-closed guard as direct UI traffic ‚Äî if you can reach the Lighthouse UI in a browser but OAuth callbacks fail with no `OAuthController` log line, check `AllowedOrigins` against the public origin first.

## Tests

### 2026-06-19 ‚Äî a WAF test that drives a request to the end of the pipeline hits `UseSpa` ‚Üí throws when `wwwroot/index.html` is absent in the backend test runner
- **Symptom**: `Verify Backend / backend` failed (green locally) on `ForwardedHeadersIntegrationTest.ForwardedHeaders_NoProxyDeclared_DirectAccessByteIdentical` (+ siblings): `System.InvalidOperationException : The SPA default page middleware could not return the default page '/index.html' because it was not found, and no other middleware handled the request.` thrown from a test `IStartupFilter` that called the downstream pipeline (`await proceed()`).
- **Root cause**: the `Verify Backend` job runs `dotnet test` against source ‚Äî there is **no built SPA**, so `Lighthouse.Backend/wwwroot/index.html` does not exist. Any integration test whose request flows to the *end* of the real `Program` pipeline reaches `app.UseSpa(...)`, which throws when `index.html` is missing. It passed locally only because a dev build had populated `wwwroot/`. The `Verify *Build` E2E jobs use the published bundle (SPA present), so they never reproduce it ‚Äî only `Verify Backend` does.
- **Fix**: the forwarded-headers test harness (`ForwardedHeadersTestStartupFilter`) no longer calls `proceed()` to observe the post-`UseForwardedHeaders` request; it invokes the real `ForwardedHeadersMiddleware` inline with the production `IOptions<ForwardedHeadersOptions>` from DI, then terminates and writes the observed scheme/host ‚Äî never touching routing/SPA/DB.
- **Rule going forward**: a `WebApplicationFactory<Program>` test must NOT let a request run to the end of the real pipeline (it hits `UseSpa` ‚Üí 500 in `Verify Backend`, which has no `wwwroot/index.html`). Hit an API endpoint that short-circuits before the SPA fallback, or invoke the specific middleware under test inline. To reproduce a "green locally, red in `Verify Backend`" failure, run the suite with `wwwroot/index.html` temporarily moved away.

### 2026-06-07 ‚Äî Zod `z.coerce.date()` silently turns `null` into epoch (1970); backend `DateTime?` fields must be `.nullable()`
- **Symptom**: E2E run ~2-3√ó normal length, multiple flow/metrics specs (CycleTime, AgingPace, FlowEfficiency, TimeInState) failing/timing out after the Zod adoption. Unit tests were all green; only live demo data exposed it.
- **Root cause**: `FeatureSchema` modelled `startedDate`/`closedDate` as required `z.coerce.date()`, but the backend `WorkItemDto` types them `DateTime?` and serialises `null` for not-started/not-closed items. `z.coerce.date()` runs `new Date(null)` ‚Üí `1970-01-01` (a *valid* Date, so no parse error) instead of preserving `null`. The old `class-transformer` `@Type(() => Date)` passed `null` through. `BaseMetricsView`'s `closedDate === null` filter then never matched ‚Üí wrong chart data ‚Üí pages never settled ‚Üí Playwright timeouts.
- **Fix**: `startedDate`/`closedDate` ‚Üí `z.coerce.date().nullable()`; assign through with `as Date` to preserve the runtime `null` while keeping the `IWorkItem` `Date` contract that 60 consumers depend on. Regression test in `FeatureService.test.ts` ("preserve null started/closed dates"). `Feature.ts:36-37,143-146`.
- **Rule going forward**: when a backend property is `DateTime?` (or any nullable type), the Zod schema field MUST be `.nullable()` ‚Äî `z.coerce.date()` alone converts `null` to epoch silently and `z.coerce.number()`/`.string()` similarly mangle null. Check the C# DTO's nullability before writing the schema; unit-test mocks won't catch it because they always supply the value. Proper follow-up: widen `IWorkItem.startedDate/closedDate` to `Date | null` and null-guard the chart consumers, removing the `as Date`.

### 2026-06-07 ‚Äî `.optional()` ‚âÝ `.nullable()`: nullable backend strings (`string?` ‚Üí JSON `null`) hard-fail the parse
- **Symptom**: after the date fix above, a *second* E2E pass still failed 5 specs ‚Äî `DeliveryMetrics`, `PortfolioDetail` (create delivery), 3√ó `TeamsDetail` forecast specs. Delivery/forecast feature grids came up empty (no rows ‚Üí checkbox/`getByText` timeouts). Flow/metrics specs passed.

### 2026-05-18 ‚Äî `IntegrationTestBase` is `[NonParallelizable]` (SQLite file-DB race in `EnsureDeleted` + `EnsureCreated`)

- **Symptom**: `Verify Backend` job on run 26051082940 failed with 1 / 2537 tests: `ProjectsControllerAuthorizationTests.UpdateProject_AsNonPremiumUser_AboveLimit_Returns403` ‚Äî `SetUp : Microsoft.Data.Sqlite.SqliteException : SQLite Error 1: 'table "AppSettings" already exists'` thrown from `IntegrationTestBase.Init()` at the `DatabaseContext.Database.EnsureCreated()` call.

### 2026-06-16 ‚Äî backend-test-speed (#5258): per-fixture isolation replaces `[NonParallelizable]`; a guard enforces the allowlist

- **What changed**: Slices 02‚Äì03 fixed the shared-state root causes the 2026-05-18 entries opted out of, so `[NonParallelizable]` was REMOVED from `IntegrationTestBase` (+ ~41 WAF fixtures), the two `*UpdaterTest`s, the 3 `*GoldTest`s, `TerminologySeederTests`, and the `*MetricsServiceTests`. For a `BackgroundService.StartAsync` test, never verify the mock immediately or with a fixed `Task.Delay` ‚Äî poll via `WaitUntilVerified`. ### 2026-05-16 ‚Äî typescript:S7735: `!== undefined` / `!== null` ternaries are flagged as "unexpected negated condition"

- **Root cause**: SonarCloud's TypeScript analyzer counts `!==`/`!=` as a negation in the condition position of a ternary `cond ? a : b`. - **Rule going forward**: Do not write `cond !== X ? truthy : falsy` ternaries ‚Äî flip to `cond === X ? falsy : truthy`. Applies to `!==`, `!=`, and `!` prefixes on the condition. Spread-or-omit conditional-spread patterns must use `=== undefined ? {} : { key: value }`, never the negated form. The rule does NOT fire on `if (x !== undefined) { ... }` guards (only on ternaries).

### 2026-05-20 ‚Äî Resolved: `Refresh Features` flake was TWO cascading races, both fixed (3 slices over the day)
- **Symptom**: `verifysqlite` and `verifypostgres` intermittently flaked Playwright tests on `tests/specs/portfolios/PortfolioDetail.spec.ts:50:37` ("should show correct Features for Jira portfolio on refresh") and `tests/specs/ado/ado.spec.ts:7:5` ("should be able to handle a team and portfolio defined in Azure DevOps"). The Refresh Features button stayed `disabled tabindex="-1"` past the 30s timeout. Initially appeared verifysqlite-only (2026-05-17 runs `25987028422`, `25991237231`), then escalated to alternating-both / both / either (2026-05-20 runs `26116104462`, `26154669070`, `26160730491`, `26162111484`). Resolved on run `26163927680` (HEAD `76bec972`): both jobs green, zero `DbUpdateConcurrencyException` in either log.
- **Root cause (turned out to be TWO races, not one)**:
  1. **Race A ‚Äî `RemoveOrphanedFeatures` in `PreprocessDataBeforeSave`**: ran on EVERY `SaveChangesAsync`, iterating the entire change tracker and marking Features with empty in-memory `Portfolios` collections as Deleted. A tiny `RefreshLog` write at the tail of `PortfolioUpdater.Update` would re-run the orphan sweep against 100+ entities accumulated across the whole update flow. Concurrent UPDATEs from other scopes (HTTP requests, periodic ticks) landing on rows the sweep had just deleted produced "expected 1 row affected, got 0" ‚Üí exception ‚Üí `SaveWithRetry` cycled 3 attempts (each ~10s) ‚Üí bubble ‚Üí 2-3 min hang in `Active updates: 1` ‚Üí Playwright 30s timeout. Removed the method entirely from the save pipeline (commit `994ac67b`); replaced with a dedicated `IOrphanedFeatureCleanupService` invoked at refresh tail + startup (commits `c7a45b09`, `ce4ce866`). Method was a janitorial sweep that had no business running on every save.
  2. **Race B ‚Äî HTTP `DELETE /api/latest/portfolios/{id}` racing the in-flight queue task**: even after Race A was fixed, `verifypostgres` still flaked. `PortfolioController.DeletePortfolio` called `portfolioRepository.Remove` synchronously in the HTTP request scope, which triggered `PortfolioRepository.Remove ‚Üí RemoveOrphanedFeatures (a SECOND code path) ‚Üí Features.RemoveRange`. Meanwhile the queue task was mid-update on the same or related portfolio. The HTTP scope's DELETE committed; the queue scope's next save found rows gone ‚Üí same exception, same 2-3 min hang. Fixed by routing DELETE through `IUpdateQueueService.EnqueueAndAwaitAsync` (new primitive added in commit `dad74e8b`, controller rewired in `8b187e8a`). The queue's single-threaded channel reader now serialises DELETE with every other queued operation on the same UpdateKey.
- **WRONG assumption that misled the first two attempts**:
  - "SQLite serialises writers, so the race rarely surfaces there" ‚Äî FALSE. SQLite's writer lock prevents corruption, but EF Core's "rows affected = 0" check still fires on the loser's UPDATE when the winner's DELETE has already committed. Counted exceptions in the same run: SQLite 23, Postgres 4. SQLite isn't race-safe for this pattern.
  - "Initial RCA pointed at orphan-sweep alone" ‚Äî partially correct. Removing the per-save sweep fixed Race A but exposed Race B which the sweep had been masking.
- **Three failed mitigation attempts before the right fix**:
  1. `ConcurrentDictionary.TryAdd` dedupe in `UpdateQueueService.EnqueueUpdate` (commit `982017c6`) ‚Äî killed the "Active updates: 2" symptom but not the underlying race. KEPT as a defence-in-depth.
  2. `entry.Collection(f => f.Portfolios).IsLoaded` guard on `RemoveOrphanedFeatures` ‚Äî regressed 5 `DeliveriesControllerIntegrationTest` cases (HTTP 404s on PUT) because `IsLoaded` is NOT set by EF relationship fixup; an Unchanged Feature whose link was removed via `Features.Clear()` is indistinguishable from a Feature loaded without `Include(f => f.Portfolios)`. Reverted in commit `a1e0d7a7`.
  3. After removing the sweep, the race persisted on `verifypostgres` ‚Äî that's when Race B was identified.
- **Architecture rules going forward**:
  1. **Do not put janitorial / cleanup logic inside `SaveChangesAsync`.** A pass that mutates the change tracker for entities beyond what the calling save was about creates unbounded race surface. Cleanup belongs in an explicit endpoint or a scheduled service that runs in its own scope.
  2. **`EntityEntry.Collection().IsLoaded` is not the right signal for "is this navigation collection trustworthy."** Only `Include`/`Load`/`LoadAsync` flip it. Relationship fixup populates the collection silently. If you need to tell "deliberately empty after fixup" from "never loaded," use the change tracker on the join entity (many-to-many) or the FK shadow property (one-to-many) ‚Äî never `IsLoaded`.
  3. **Any HTTP write-path that races a queued update on the SAME entity MUST go through the queue.** The queue is the canonical "exclusive operation on entity X" primitive ‚Äî it already exists, it already serialises, use it. Adding a sibling semaphore parallel to the queue duplicates the abstraction. For one-shot synchronous handlers that need to wait on completion (e.g. controller methods that historically returned 200/204 synchronously), use the new `IUpdateQueueService.EnqueueAndAwaitAsync(...)` primitive (added in `dad74e8b`) which returns a `TaskCompletionSource`-backed `Task` that completes when the inner work runs.
  4. **A test that's silently masked by a per-save sweep is an accident waiting to happen.** Before Race A was fixed, `RbacExceptionEndpointsAuthorizationTests.SeedFeatureAsync` had a latent `NOT NULL Features.Order` violation that only surfaced once the sweep stopped silently detaching the Added entity. When removing any "defensive" code path, run the full test suite ‚Äî failures that previously couldn't surface will now expose themselves.
- **Known follow-ups** (still open):
  - `TeamController.Delete` calls `TeamRepository.Remove` which has the same shape as `PortfolioRepository.Remove` (cascade-delete via OrphanedFeatures). Same race pattern. Separate slice when prioritised. Until then, the verifypostgres flake risk for team-delete-during-update remains.
  - `UpdateQueueService` is a single global channel; different `UpdateKey`s still serialise sequentially. Acceptable for current workload but could be a throughput cap as portfolios scale. Marked as `[Ignore]`'d test in `PortfolioDeleteSerialisationTests.DeletePortfolio_DifferentPortfolioInFlight_DoesNotBlock` with documented reason.

### 2026-06-02 ‚Äî csharpsquid:S2139 (recurrence): a domain-event handler must not log-and-rethrow when the dispatcher already logs-and-swallows
- **Symptom**: `sonar-gates` `new_violations` on `LetPeopleWork_Lighthouse` flagged `csharpsquid:S2139` at `DeliveryMetricSnapshotRecordingHandler.cs:40` ‚Äî "Either log this exception and handle it, or rethrow it with some contextual information." The catch block did `logger.LogError(exception, ...); throw;`. SonarCloud-only; `dotnet build` was clean.
- **Root cause**: log-AND-rethrow in the same catch is the S2139 trigger (duplicate log entries: once here, once wherever the rethrow is finally caught). Decisive context: `DomainEventDispatcher.InvokeHandlerSafely` ALREADY wraps every handler call in `try { await handler.HandleAsync(...) } catch (Exception ex) { logger.LogError(ex, "Handler {HandlerType} failed ...; continuing with remaining handlers"); }` (with a deliberate `#pragma warning disable CA1031` and the rationale "one failing handler must not abort the others or lose the committed fact; recovery is the next re-sync"). So the handler's `throw;` never reached `ForecastUpdater`/`PortfolioUpdater` (the dispatcher swallows it) AND the dispatcher already logged it ‚Äî the handler's LogError was redundant and the rethrow was dead.
- **Fix**: chose **log-and-handle (swallow)** ‚Äî kept the handler's contextual `LogError` (it carries the Portfolio id + elapsed-ms that the dispatcher's generic log lacks), removed the `throw;`, and reworded the message to state snapshot recording is best-effort and the next forecast update retries. **This is a behaviour change at the handler unit level** (it no longer throws on persistence failure) but NOT at the system level (the dispatcher already swallowed it). The handler unit test that asserted rethrow (`HandleAsync_WhenSnapshotPersistenceFails_PropagatesTheException`, `Throws.InstanceOf<InvalidOperationException>`) was updated to `HandleAsync_WhenSnapshotPersistenceFails_SwallowsAndLogsTheError`: the call now completes without throwing AND the test `logger.Verify(...)`s a single `LogLevel.Error` with the original exception.
- **Rule going forward**: before writing a `try { ... } catch { logger.Log*(ex, ...); throw; }` in a domain-event handler, check the dispatcher: if `IDomainEventDispatcher.PublishAsync`/`InvokeHandlerSafely` already catches-logs-and-does-not-propagate (it does ‚Äî by design, so one handler failing can't abort the others or lose the committed fact), then the handler's catch must EITHER (a) swallow with a contextual `LogError` and NO `throw;` (best-effort handlers like metric/snapshot recording ‚Äî preferred, this entry), OR (b) not catch at all and let the dispatcher log it. Never both log AND rethrow (S2139), and never `throw new ...(..., ex)` to "add context" when the dispatcher is the terminal catch ‚Äî that just changes the type the dispatcher logs, adding nothing. When you flip a handler from rethrow to swallow, you MUST update the unit test that pinned the rethrow to assert swallow-and-log instead (do not delete the test ‚Äî it still guards that failures are observable in logs).

### 2026-06-05 ‚Äî verifypostgres "column does not exist": a new NOT NULL column with no Postgres default fails the restore-old-backup E2E
- **Symptom**: `verifypostgres` fails deterministically (verifysqlite + backend + sonar all green, and a clean full-pipeline re-run fails identically); app errors with `Npgsql 42703: column t.WaitStates / p.WaitStates does not exist` during an MVC request ‚Äî NOT at startup ‚Äî even though the migration `.cs` + `.Designer` + `ModelSnapshot` are all committed, correct, and present in the built/published DLL.
- **Root cause**: the `DatabaseManagement.spec.ts` E2E (`Should be able to restore and create a new backup`) restores a committed OLD backup fixture `tests/fixtures/Lighthouse_Backup_2026-03-24_{postgresql,sqlite}.zip` ‚Äî a dump with POPULATED Teams/Portfolios from before the new column existed. `DatabaseManagementService.RestoreBackup` then calls `MigrateAndSeedDatabase()`, re-running migrations onto the restored (populated) schema. The Postgres `AddWaitStates` migration did `migrationBuilder.AddColumn<List<string>>(... nullable: false)` with NO default ‚Üí `ALTER TABLE "Portfolios" ADD "WaitStates" text[] NOT NULL;` which Postgres REJECTS on a non-empty table (`ERROR: column "WaitStates" of relation "Portfolios" contains null values`). The migration throws, the column is never created, and every later query says "column does not exist". verifysqlite passed because the SQLite `AddWaitStates` migration was scaffolded WITH `defaultValue: "[]"`; the Postgres one (a `List<string>`‚Üí`text[]` primitive collection) was scaffolded WITHOUT a default. On the app's first startup the DB is empty so the column adds fine ‚Äî the failure only surfaces after the restore re-applies the migration to populated tables.
- **Fix**: hand-edit the Postgres migration to give the column a default: `AddColumn<List<string>>(... nullable: false, defaultValueSql: "ARRAY[]::text[]")`. **`defaultValue: new List<string>()` does NOT work** ‚Äî an empty-list default emits no SQL `DEFAULT` clause; use `defaultValueSql` so the generated DDL is `ADD "WaitStates" text[] NOT NULL DEFAULT (ARRAY[]::text[])`, which succeeds on populated tables. No ModelSnapshot/Designer change needed (a migration-only default does not alter the model). Mirrors the SQLite migration's `defaultValue: "[]"`.
- **Rule going forward**: every NEW `nullable: false` column added to a table the backup-restore E2E populates (Teams, Portfolios, and friends) MUST have a default in BOTH provider migrations, because restore re-applies migrations onto OLD populated data. For Postgres array columns (`List<string>` ‚Üí `text[]`) the EF scaffolder OMITS the default ‚Äî always hand-add `defaultValueSql: "ARRAY[]::text[]"` (an empty `List<string>()` default is a no-op). A green `verifysqlite` does NOT imply `verifypostgres`: SQLite scaffolds a `defaultValue`, Postgres does not. If `verifypostgres` alone fails with `42703 column ... does not exist` for a committed migration, check the migration for a missing Postgres default BEFORE suspecting build/DLL issues ‚Äî the column add silently failed on the restore step's populated tables.
