## Summary

Closed a semantic defect in `Lighthouse.Backend/Factories/IssueFactory.cs` and `Lighthouse.Backend/Services/Implementation/WorkTrackingConnectors/AzureDevOps/AzureDevOpsWorkTrackingConnector.cs` that caused work items following the path `ToDo → Doing(T1) → ToDo → Done(T3)` to keep `StartedDate = T1` instead of resetting it to `T3`. The expected behaviour — and the one already enforced for the direct `ToDo → Done` path via the existing null-fallback — is that an item which never re-entered Doing on the path to Done has `StartedDate == ClosedDate` (it was never truly "in progress" at the moment it closed).

Customer Liz reported the defect against Jira (ADO Bug 5064) and noted the equivalent expectation in ADO and Linear. The fix lands in Jira + ADO; Linear is **out of scope** because Linear exposes a single mutable upstream `startedAt` field that cannot be reconstructed from a workflow changelog without a richer GraphQL `issueHistory` query — a strictly larger change deserving its own bug if customer evidence demands.

## Business context

Reported via ADO Bug 5064 (created 2026-05-21 19:28 UTC, activated 2026-05-21 19:31 UTC by Benj). Reporter (Liz) is using Lighthouse against Jira. The bug undermines CycleTime / WorkItemAge derived metrics on `WorkItemBase` (lines 46-84) for any item that bounces in and out of Doing before closing — a common pattern when a team triages an item, defers it back to the backlog, and later closes it as "won't fix" without re-doing implementation. For those items, the reported CycleTime included the originally-planned implementation window even though the actual work never resumed.

Liz's exact phrasing on the ADO bug: "*I think that's working already like this if it never goes to in progress*". That observation correctly pointed at the existing null-fallback (`IssueFactory.cs:79-83` / `AzureDevOpsWorkTrackingConnector.cs:743-746`) — which patches `startedDate = closedDate` when `startedDate == null && closedDate != null` — and made the fix shape obvious: extend the SAME semantics to the revert-then-skip-Doing equivalence class by invalidating `startedDate` when the most recent ToDo-revert post-dates it.

## Carry-forward / follow-ups

- **Shared `StateTransitionDateResolver` refactor**: the algorithm is now duplicated in three places — `IssueFactory.GetStartedAndClosedDate`, `AzureDevOpsWorkTrackingConnector.GetStartedAndClosedDateForWorkItem`, and (in a different shape) `LinearWorkTrackingConnector` — and each carries an identical Done-branch / ToDo-invalidation / null-fallback triplet. Extracting a connector-agnostic policy that takes a history-iterator and returns `(StartedDate, ClosedDate)` would (a) eliminate the duplication, (b) make Linear's missing per-revert handling visible at the call site, (c) make future invariant additions (e.g. enforce `StartedDate <= ClosedDate` always) a one-line change. Deferred per CLAUDE.md "refactor commit separate from fix commit". File as a `refactor(work-items)` work item if there's appetite.
- **Linear connector support**: Linear's `startedAt` GraphQL field is a single mutable upstream value. To match Jira / ADO semantics on revert-then-skip-Doing, Lighthouse would need to query Linear's `issueHistory` for state-change events and apply the same invalidation rule client-side. File a new bug if customer evidence shows the Jira / ADO discrepancy reproduces against Linear. **No follow-up scheduled** — wait for evidence.
- **ADO regression test for fixture work item 400 (or next free ID)**: The existing ADO test class is a real-IO integration test against `CMFTTestTeamProject`; adding a regression for bug 5064 requires provisioning a real work item with the `ToDo → Doing → ToDo → Done` history. Deferred because (a) fixture provisioning wasn't feasible in-session, (b) the Jira test covers the shared semantic, (c) customer instances self-heal on next sync. **Recommended follow-up**: when next touching the ADO test fixtures, add work item 400 and the corresponding test method `SetStartedAndClosedDate_ItemMovedFromToDoToDoingBackToToDoThenToDone_StartedEqualsClosed` modelled on `AzureDevOpsWorkTrackingConnectorTest.cs:287`.
- **Pre-existing untested helpers in `IssueFactory.cs`**: Stryker exposed 17 surviving mutants in `GetRankFromFields` (lines 98-119), `TryGetRankFromRankField` (lines 122-137), and `GetLabelsFromFields` (lines 151-164). These are pre-existing test coverage gaps unrelated to bug-5064. If the team wants to lift the file-wide kill rate, that work belongs in its own `test(work-items)` slice. Not blocking.
- **Invariant guard on `WorkItemBase`**: a `[Test]` or production-side guard asserting `StartedDate == null || ClosedDate == null || StartedDate <= ClosedDate` would catch a wider class of date-derivation bugs at the boundary. Useful but orthogonal; not scoped here.
