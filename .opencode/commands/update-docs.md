# /update-docs — keep docs and screenshots in sync

You are the on-duty doc minder. Goal: after a chunk of UI/backend work, make sure `docs/**.md` still describes what the product does, and that every `docs/assets/**.png` reflects the current UI. Output: an approved diff of test changes, a re-run of the `@screenshot` suite, and a list of which images actually changed.

## Fixed context (do NOT ask)

- **Docs root**: `docs/` — Jekyll site, markdown files under topical folders (`Installation/`, `concepts/`, `feature/`, `metrics/`, `settings/`, `teams/`, `portfolios/`, `licensing/`, `compliance/`, `faq/`, `releasenotes/`, `aiintegration.md`, `index.md`).
- **Assets root**: `docs/assets/` — subfolders mirror the topic areas (`authentication/`, `concepts/`, `features/`, `installation/`, `licensing/`, `settings/`, plus `features/metrics/`). Every `*.png` here is regenerated by a `@screenshot` test.
- **Single screenshot spec**: `Lighthouse.EndToEndTests/tests/specs/screenshots/Screenshots.spec.ts` — all `@screenshot`-tagged tests live here. Add new tests in the same file unless the file gets unwieldy (>800 lines); prefer many small focused tests over big ones so each PNG can be regenerated independently.
- **Helpers**: from `tests/helpers/screenshots.ts`:
  - `takePageScreenshot(page, "<subfolder>/<file>.png")` — full-page screenshot.
  - `takeDialogScreenshot(locator, "<subfolder>/<file>.png")` — scoped to a locator (dialog, card, panel).
  Both resolve relative paths under `getPathToDocsAssetsFolder()` → `docs/assets/`. **Smart write**: the helper only overwrites the on-disk PNG if pixel-diff exceeds `maxDiffPercentage` (default 0.5%). Re-running is therefore safe and idempotent — only genuinely-changed images touch disk.
- **Test fixtures**: `overviewPage`, `testWithData(...)`, plus `LighthouseFixture` exports — read the top of `Screenshots.spec.ts` for the import set in use; reuse the same fixtures so new tests start from the same seeded state.
- **Backend port**: `5169` (HTTP) — Playwright's default `baseURL` (`TestConfig.LighthouseUrl`) targets `http://localhost:5169/`. Use the `http` launch profile.
- **Database**: SQLite, files `Lighthouse.Backend/Lighthouse.Backend/LighthouseAppContext.db{,-shm,-wal}`. Always start from a clean DB so seeded test state is deterministic. The repo ships `Lighthouse.Backend/Remove-LighthouseDb.ps1` for this, but it's interactive (PowerShell `Read-Host`). On Linux without `pwsh`, delete the files directly with `rm`.
- **Test runner**: from `Lighthouse.EndToEndTests/`, `pnpm screenshots` (= `playwright test --grep @screenshot`). Reporter is `html`; results land in `Lighthouse.EndToEndTests/playwright-report/`.
- **Image reference patterns in docs**:
  - Relative: `assets/<topic>/<file>.png` (docs pages — resolved by Jekyll).
  - Absolute: `https://raw.githubusercontent.com/LetPeopleWork/Lighthouse/refs/heads/main/docs/assets/<topic>/<file>.png` (release notes).
  Both forms reference the same on-disk PNGs, so regenerating the PNG fixes both.

`$ARGUMENTS` may optionally be a git ref (tag, SHA, or branch) to anchor the "since" point. If empty, default to the latest tag on `main`.

## Phase 1 — analyze what likely needs updating

Run in parallel:

```bash
git tag --sort=-creatordate | head -3
git log <anchor>..HEAD --no-merges --pretty='%h %s'
git diff --stat <anchor>..HEAD -- Lighthouse.Frontend/src docs Lighthouse.EndToEndTests/tests
git diff --name-only <anchor>..HEAD -- Lighthouse.Frontend/src
```

From that data, build three lists:

1. **Frontend areas that changed** — group changed files in `Lighthouse.Frontend/src/` by top-level folder (`components/`, `pages/`, `models/`, `tests/`, etc.). The folder name usually tells you which doc topic is affected (`pages/Overview/*` ↔ `docs/concepts` + landing screenshots; `pages/Metrics/*` ↔ `docs/metrics`; `pages/Settings/*` ↔ `docs/settings`; `pages/TeamDetail/*` ↔ `docs/teams`; `pages/PortfolioDetail/*` ↔ `docs/portfolios`).
2. **Docs pages that may be stale** — for each changed frontend area, find the matching doc(s):
   ```bash
   grep -rli "<feature keyword>" docs/ --include="*.md"
   ```
   Read the matched pages. Flag anything that contradicts the new behaviour (renamed control, removed step, new section needed, removed feature still documented). Pure prose drift counts; this command isn't *only* about screenshots.
3. **Screenshot coverage** — read `Screenshots.spec.ts`. For each test, note the `takePageScreenshot` / `takeDialogScreenshot` path it writes to. Build a map `<docs/assets path> → <test title>`. Cross-reference with:
   ```bash
   git diff --name-only <anchor>..HEAD -- docs/assets
   ```
   This tells you which PNGs were last touched by hand (often: new images committed without a generating test). Each one is a candidate for a new test.
4. **Under-illustrated doc pages** — independent of recent changes, look for docs pages that *describe* UI but ship no screenshots. For each markdown file in `docs/`:
   ```bash
   for f in $(find docs -name "*.md" -not -path "docs/_site/*" -not -path "docs/releasenotes/*"); do
     img_count=$(grep -cE '(\!\[.*\]\(.*\.(png|jpg|jpeg|svg)\)|src=".*\.(png|jpg|jpeg|svg)")' "$f")
     ui_words=$(grep -ciE '(click|button|tab|dialog|wizard|page|section|field|dropdown|toggle|panel)' "$f")
     [ "$img_count" = "0" ] && [ "$ui_words" -gt 3 ] && echo "$f  (0 images, $ui_words UI mentions)"
   done
   ```
   That heuristic catches pages that talk about UI actions but have no visual anchor. Read each match — some will be legitimately text-only (conceptual explainers, reference tables, FAQ entries) and should be left alone. The rest are **coverage gaps**: candidates for a new `@screenshot` test to add a single illustrative image to. Cross-reference with the map from step 3: if a doc page has zero images AND no screenshot test ever wrote to a matching `docs/assets/<topic>/` path, it's a strong candidate. Carry into the change matrix as `type: coverage-gap, action: propose new test`.

Produce a **change matrix** in memory with one row per candidate:

| Candidate | Type | Action | Evidence |
| --- | --- | --- | --- |
| `docs/assets/features/teamdetail.png` | existing | rerun | `Lighthouse.Frontend/src/pages/TeamDetail/QuickSettingsBar.tsx` changed |
| `docs/teams/edit.md` | doc page | prose update | RBAC tabs renamed but doc still says "Settings tab" |
| `docs/assets/settings/rbac/<new>.png` | new | new test | new RBAC settings tab, no PNG, no test |
| `docs/assets/concepts/oldonboarding.png` | existing | retire | feature removed in `<sha>`, image still referenced from one doc page |
| `docs/settings/configuration.md` | coverage-gap | propose new test | 0 images, 14 UI mentions ("click", "tab", "section") |

Be conservative about "retire" — only flag when you have direct evidence the feature is gone (file deletion, removed route, removed component) AND the image is no longer referenced from any markdown file.

Be conservative about "coverage-gap" too — concept/reference pages can legitimately be text-only. A page belongs on this list only if it walks the reader through concrete UI actions ("open Settings → Configuration, click Add…") with no image to anchor them.

## Phase 2 — confirm scope with user

Use `question`. Compose questions so the user can quickly OK/edit the matrix. Group by action (each is its own `question` — call them together, up to 4 per call):

1. **Re-run only** (multi-select): list candidate existing PNGs you'd regenerate without writing new tests. Default-select all — these are safe (pixel-diff guard means no-ops won't touch the file). User can deselect any they want to skip (e.g. unchanged metrics screenshots when only the overview changed).
2. **New tests — change-driven** (multi-select): list each new `@screenshot` test you'd add *because of a code change since the anchor*. Include the target path and a one-line description. User picks which to actually write. For anything not on this list, do not invent.
3. **Existing tests to extend** (multi-select): list tests that need code changes (e.g. an existing big test needs an extra `takeDialogScreenshot` for a new sub-element) vs ones that already exist and just need rerunning. Each pick maps to a planned diff.
4. **Doc prose changes** (multi-select): list markdown files where you want to edit prose, with a one-line summary of the change. The user can defer prose edits to a separate session if they prefer.

**Coverage-gap proposals** (separate `question` call, only if Phase 1 step 4 produced any rows): list each under-illustrated doc page with a one-line proposal — *which* page state to screenshot and *which* `docs/assets/<topic>/<file>.png` path to write to. Default-deselect all; user opts in. These are suggestions, not change-driven obligations. Phrase the question explicitly as "Optional: add screenshots to these docs pages that have none today?" so it's clear the user can say no without it implying anything is broken.

Don't ask whether to delete a doc/screenshot unless the matrix has a "retire" row — and even then, propose the deletion explicitly with the evidence, do not auto-delete.

If the user picks "Other" or rejects everything, exit early with a summary — there is nothing to execute.

## Phase 3 — execute test/doc updates

Apply edits in this order, one approved item at a time:

**Prose-only doc edits**: straightforward `edit` / `write`. Match the existing tone (terse, present tense, no marketing fluff). If the change references a screenshot path that doesn't exist yet, keep the link — Phase 4 will create it.

**Extend an existing screenshot test**: add a `takePageScreenshot` / `takeDialogScreenshot` call inside the relevant `test(...)` block. Choose the target path under the right `docs/assets/<topic>/` subfolder using `kebab-or-camelCase.png` matching local conventions (look at sibling filenames). Don't bundle unrelated screenshots into one test — if the new shot needs its own page-state setup, it gets its own test.

**Write a new screenshot test**: place it in `Screenshots.spec.ts`, mirroring the existing tests' style. Default skeleton:

```ts
test("Take @screenshot of <thing being captured>", async ({ overviewPage }) => {
    // navigate to the relevant page state via the page-object model
    const targetPage = await overviewPage.lightHousePage.goTo<...>();
    // any setup (open dialog, click tab, seed data)
    await takePageScreenshot(targetPage.page, "<topic>/<file>.png");
});
```

- Pick the right fixture: `overviewPage` for tests that start from the seeded overview state; `testWithData(...)` if the test needs specific work-tracking-system / team / portfolio identifiers wired in (read the existing `testWithData(...)` blocks for the pattern).
- Use `takeDialogScreenshot(locator, "...")` when you only need a dialog or panel, not the full page. It produces tighter, more stable images.
- Prefer **many small tests** over one big one. Each test owns exactly one or two PNGs. This matches the existing pattern and lets `pnpm screenshots --grep "<title fragment>"` regenerate just one image when needed.
- Re-use page-object methods from `tests/models/` rather than raw selectors. If a page-object method is missing, extend the page object (and only the page object) — do not inline selectors in the test.
- No hard `waitForTimeout`. Use Playwright's auto-waiting (`expect(...).toBeVisible()`, `waitForResponse`, etc.) and let the screenshot helper's `waitTimeInMilliSeconds` handle final settling.

After every edit, run the local quality gate on the changed files:

```bash
cd Lighthouse.EndToEndTests && pnpm dlx @biomejs/biome check --write ./tests
```

(That's the `prebuild` hook — running it now catches lint/format issues before Phase 4.)

## Phase 4 — clean DB, start backend, re-run screenshots

Sequential because each step depends on the prior one.

**Step 4a — stop any running backend on 5169** (don't double-start):

```bash
lsof -tiTCP:5169 -sTCP:LISTEN 2>/dev/null | xargs -r kill
sleep 1
lsof -tiTCP:5169 -sTCP:LISTEN 2>/dev/null  # should be empty
```

If something is still listening after that, stop and tell the user — do not force-kill someone's dev instance without consent.

**Step 4b — clean the database**:

```bash
ls Lighthouse.Backend/Lighthouse.Backend/LighthouseAppContext.db* 2>/dev/null
rm -f Lighthouse.Backend/Lighthouse.Backend/LighthouseAppContext.db{,-shm,-wal}
```

(The shipped `Remove-LighthouseDb.ps1` is interactive PowerShell — only use it if `pwsh` is available AND you can stream stdin; otherwise the `rm` is the documented behaviour of that script anyway.)

**Step 4c — start the backend in the background**:

```bash
dotnet run --project Lighthouse.Backend/Lighthouse.Backend/Lighthouse.Backend.csproj --launch-profile http
```

Launch in background. Then poll readiness (do NOT sleep blindly):

```bash
until curl -sf http://localhost:5169/health >/dev/null 2>&1 || curl -sf http://localhost:5169/ >/dev/null 2>&1; do sleep 2; done
```

If `/health` doesn't exist, the root `/` returning 200 is fine. Cap the wait at ~90 seconds via a `timeout` wrapper; if it doesn't come up, surface the dotnet logs and stop.

**Step 4d — run the screenshot suite**:

```bash
cd Lighthouse.EndToEndTests && pnpm screenshots
```

If only a subset is in scope (Phase 2 chose specific tests), grep for them so you don't spend 5+ minutes on the whole suite:

```bash
cd Lighthouse.EndToEndTests && pnpm exec playwright test --grep "@screenshot" --grep "<title fragment>"
```

`pnpm exec playwright test --grep '@screenshot' --grep '<title fragment>'` is fine for one-off filters; the package script `pnpm screenshots` is the full run.

If a test fails (not a pixel-diff — those are silent updates — but an actual Playwright failure), capture the failing test name and the first ~50 lines of the trace from `playwright-report/`. Do not retry-loop. Tell the user; they may need to update a page-object selector that drifted.

**Step 4e — stop the backend**:

```bash
lsof -tiTCP:5169 -sTCP:LISTEN | xargs -r kill
```

If you started the dotnet process in background, kill that process group too. Always clean up — leftover dotnet processes block the next run.

## Phase 5 — final report

```bash
git status --short docs/assets Lighthouse.EndToEndTests/tests/specs/screenshots docs
git diff --stat docs/assets
```

Summarize in ≤6 lines:

1. Anchor + commit count in window.
2. Tests written / extended (count + filenames).
3. Doc prose edits (count + filenames).
4. PNGs actually changed (count — from `git status docs/assets`). Smart-write means many will be no-ops; that's expected and good.
5. Any test failures or skipped items, with a 1-line reason each.
6. Suggested next step: `git diff docs/assets | head` to spot-check one PNG visually, or `git restore docs/assets/<file>.png` to revert any unintended change.

Do NOT commit or push. The user reviews the diff.

## Guardrails

- Never run `pnpm screenshots` against a backend you didn't start (or a dirty DB). The whole point is determinism — running against the user's dev instance will produce noisy diffs.
- Never delete a screenshot file directly. Only the `rm Lighthouse.Backend/.../LighthouseAppContext.db*` is allowed without user confirmation; everything in `docs/` requires the user's explicit OK.
- Never disable a `@screenshot` test to make the run green. If it fails, fix the page-object or surface for human review.
- Never silently widen `maxDiffPercentage` in `helpers/screenshots.ts` to "fix" a flaky pixel diff — tune wait time or assertion order in the test instead.
- If `pwsh` is available and the user prefers using `Remove-LighthouseDb.ps1`, that's fine — but pipe `'y'` to it (`echo 'y' | pwsh -File Lighthouse.Backend/Remove-LighthouseDb.ps1`) so it doesn't block on `Read-Host`.
- If Phase 4 surfaces unrelated failing E2E tests (not in the @screenshot grep), do not chase them — note them and move on. This command is scoped to docs/screenshots.
- Don't write CSS-/style-only tests or "assert renders" placeholder tests. Every new `@screenshot` test must end in at least one `takePageScreenshot` / `takeDialogScreenshot` call against a deterministic page state.
